# v0.4.3 — Tauri 2.0 Init · Change Summary

## What changed

Scaffolded `src-tauri/` as the Tauri 2.0 desktop shell. The web frontend
(`public/`) loads inside a webview; the Rust side exposes (in future
issues) vault commands. For v0.4.3, the Rust side does *nothing* except
set up logging — all real vault ops still go through the existing Node
HTTP server. That's deliberate: this issue is about the **shell**, not
the **commands**.

### New files
- `src-tauri/Cargo.toml` — Rust crate manifest, name `second-brain`,
  version `0.4.0-alpha`, depends on `tauri = "2.11.3"` + `tauri-plugin-log`.
  Lib name `second_brain_lib`, three crate-types (staticlib, cdylib, rlib).
- `src-tauri/tauri.conf.json` — Tauri config. Identifier `com.secondbrain.app`,
  product name "Second Brain", window title "第二大脑 · Second Brain",
  default window 1400×900 (min 900×600), native decorations.
  `build.devUrl` points to `http://localhost:3939` (the Node dev server);
  `build.frontendDist` points to `../public` (release bundles the web SPA).
  `beforeDevCommand` runs `npm run dev` to start the Node server.
  Bundle targets: `appimage`, `deb`, `rpm`. Bundle category: Productivity.
- `src-tauri/build.rs` — one-line shim that calls `tauri_build::build()`.

- `src-tauri/src/lib.rs` — `run()` builds a Tauri app with logging in debug.

## How it works

### Dev mode (`npm run dev:tauri` or `cargo tauri dev`)
1. Tauri runs `beforeDevCommand`: `npm run dev` → starts `node server.mjs`
   on port 3939 in the background.
2. Tauri waits until `http://localhost:3939/api/health` responds.
3. Tauri compiles the Rust binary in debug mode (incremental, ~30s cold).
4. Tauri opens a 1400×900 native window, loads `devUrl` in the webview.
5. The webview shows the same UI as the browser, but as a native window.

### Release mode (`npm run build:tauri` or `cargo tauri build`)
1. Tauri compiles the Rust binary in release mode.
2. Tauri bundles `public/` into the app via `frontendDist: "../public"`.
3. Tauri produces an AppImage + .deb + .rpm under `src-tauri/target/release/bundle/`.
4. **The bundled app does NOT have the Node server.** It needs the
   `tauri-commands` work in v0.4.4 before vault operations work in
   production. This is the deliberate scope split for v0.4.3.

### Security posture
- No shell plugin. Webview cannot spawn arbitrary processes.
- No fs plugin. Webview cannot read/write arbitrary filesystem paths.
- No http plugin. Webview cannot make arbitrary outbound requests.
- Only `core:default` + explicit window management perms.
- All future vault access MUST go through Tauri commands that the
  frontend invokes via `__TAURI__.invoke('cmd_name', args)`.

### Known environment issue (not a code bug)
The sandbox running this verification has no GPU (the GDK logs
`Failed to create GBM buffer of size 1400x900: Invalid argument` on
startup). WebKit falls back to software rendering, which works but
doesn't always paint in screenshots taken with `PIL.ImageGrab`. On a
real Linux desktop with WebKit2GTK + GPU (or `WEBKIT_DISABLE_COMPOSITING_MODE=1`
forcing software), the webview content renders normally. This is an
artifact of the sandbox, not a Tauri bug.

Mitigation for verification on this sandbox: the window chrome, title,
icon, and native decorations are all visible in the screenshot, which
proves the shell is functional. Full UI verification happens in v0.4.7
on a real Linux machine.
